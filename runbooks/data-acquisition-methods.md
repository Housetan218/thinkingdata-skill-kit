# ThinkingData 数据获取固定办法

更新时间：2026-04-03

> 目的：把团队常用跑数路径固定为“项目确认 -> SQL -> 结果整理”，避免回到纯看板读数模式。

## 1. 固定结论

- 看板只做命名映射和口径校准
- 主取数路径是 SQL
- SQL 默认优先走 WebSocket 直连
- 新项目先探结构，再写正式 SQL
- 已建档项目不需要每次全量探表
- 没有明确查询目标时，不进入 SQL 执行

## 2. 冷启动与日常路径

### 2.1 冷启动

适用于：

- 新同事第一次接手
- 新项目
- 新子项目
- 已有模板明显失真

顺序：

1. 登录后台
2. 读取 `ACCESS_TOKEN`
3. 确认项目
4. 探用户表和事件表
5. 确认产品识别字段
6. 固化 SQL 模板
7. 后续查询转入 WebSocket 直连

### 2.2 日常跑数

适用于：

- 项目已建档
- 产品映射已确认
- 模板已存在

顺序：

1. 登录后台
2. 读取 `ACCESS_TOKEN`
3. 确认项目和日期
4. 直接跑现有 SQL 模板
5. 整理输出

不是每次都扫描全部项目，也不是每次都重做探表。

进入第 4 步前，至少要明确：

- 项目或产品范围
- 日期或日期范围
- 指标
- 输出格式

缺任何一项，就停在第 3 步，不要为了推进而打开 SQL 页。

## 3. WebSocket 直连

已验证可用的 SQL 后端路径：

```text
wss://taadmin.unbing.cn/v1/ta-websocket/query/<ACCESS_TOKEN>
```

真实发送格式：

```json
["data", { "requestId": "WS_SQLIDE@@...", "...": "..." }, { "channel": "ta" }]
```

关键字段：

- `projectId`
- `qp`
- `eventModel: 10`
- `querySource: "sqlIde"`
- `searchSource: "model_search"`
- `isVisualInitialQuery: false`
- `useCache: false`

结果读取：

- 最终 `progress=100`
- 从 `result.data.rows` 取结果
- `result.data.queryId` 是查询 ID
- `result.data.elapsedTimeMillis` 是引擎耗时

Bundled script：

```text
scripts/run_sql_via_ws.js
```

示例：

```bash
TD_ACCESS_TOKEN='<current ACCESS_TOKEN>' \
node scripts/run_sql_via_ws.js 123 /absolute/path/to/query.sql
```

## 4. 什么时候需要探表

以下场景才回到探表流程：

- 新项目
- 聚合项目拆产品
- 包名或过滤字段变化
- 查询结果和历史差异过大
- 指标属于留存、保存率、使用率等口径型指标，但模板尚未校准

优先探查：

1. `v_user_<projectId>`
2. `v_event_<projectId>`
3. 空间树
4. 报表定义

## 5. 常见错误

- 先开看板直接读值
- 聚合项目不拆产品
- 结果异常时仍然硬套旧模板
- 明明已能直连还继续围绕 SQL 页面点来点去
- 查询目标未定义就先打开 SQL 页

## 5.1 卡住场景处理

现象：

- 同事说“我已经打开 SQL 页面了”
- 但 5 到 10 分钟都没有明确 SQL、结果或报错

优先判断：

1. 是不是没有明确项目
2. 是不是没有明确日期
3. 是不是没有明确指标
4. 是不是没有明确要什么格式的结果

如果答案是“有任何一项没有”，不要继续停留在 SQL 页，直接回到任务定义层。

## 6. 性能判断

2. 服务端排队
3. SQL 引擎执行

## 5. 当前固定工作模型

### 5.1 单产品项目

适用项目：

- `101`
- `123`
- `125`
- `126`
- `131`
- `132`

默认办法：

- 先用主包值过滤或校准
- 再查新增、活跃、留存、收入、国家分布等

### 5.2 聚合项目

适用项目：

- `160`
- `170`

默认办法：

- 必须先加产品过滤条件
- 不允许直接对整个项目裸跑业务数据

### 5.3 特殊项目

适用项目：

- `171`

默认办法：

- 不先假设有 `bundle_id`
- 先扫：
  - `campaign_name`
  - `campaign`
  - `media_source`
  - `ad_name`
  - `first_install_version`

## 6. 子项目探表推荐顺序

后续统一采用“一项目一组 SQL”的探查方式，先做结构识别，再做正式查询。

### 6.1 第 1 组：用户表字段分布

目标：

- 找产品识别字段
- 找测试包/历史包
- 找投放或渠道字段

优先顺序：

1. `bundle_id`
2. `ub_app_package_name`
3. `campaign_name`
4. `media_source`
5. 其他可疑识别字段

### 6.2 第 2 组：事件表事件分布

目标：

- 找新增事件
- 找业务功能事件
- 找可能的比率分子事件

优先输出：

- 事件名 top 列表
- 目标日期事件量
- 目标日期去重用户量

### 6.3 第 3 组：口径字段探查

目标：

- 找 `test_type`
- 找 `mode`
- 找国家字段
- 找版本字段

结论要写清：

- 字段是否存在
- 在哪张表存在
- 值的分布是什么
- 是否能直接进正式 SQL

### 6.4 第 4 组：正式 SQL

只有当前 3 组探查完成后，才允许写：

- 新增 SQL
- 活跃 SQL
- 国家分布 SQL
- 留存 SQL
- 使用率/保存率 SQL

说明：

- 对于已建档项目，日常查询允许直接跳到这一组。
- 对于未建档或异常项目，必须从第 1 组重新开始。

## 7. 常见错误办法

### 7.1 错误：把看板当成主取数入口

问题：

- 页面读数快，但不稳定
- 容易受预计算、缓存、默认条件影响
- 不适合批量化

正确办法：

- 看板只做映射和校准
- 主数据直接走 `SPL/SQL`

### 7.2 错误：先看结果，再猜字段

问题：

- 很容易把多个产品混在一起
- 聚合项目最容易出错

正确办法：

- 先探查项目字段
- 先确认产品识别字段
- 再跑正式 SQL

### 7.3 错误：聚合项目不加产品过滤

问题：

- `160`、`170` 会直接串产品
- 得到的新增、活跃、ROI、留存都不可信

正确办法：

- 必须先按 `bundle_id` 或识别字段拆产品

### 7.4 错误：跳过探表，直接写业务 SQL

问题：

- 容易把测试包、历史包、聚合产品混进来
- 容易把看板里的隐藏条件误当成裸 SQL 逻辑
- 不同同事会写出不同口径

正确办法：

- 先跑固定探表 SQL
- 先登记字段和过滤值
- 再写正式业务 SQL

### 7.5 错误：把 SQL 结果表格当成稳定 DOM 抓取对象

问题：

- 前端表格结构不稳定
- 很难通用抽取

正确办法：

- 统一使用 `全量下载`
- 本地读 CSV

### 6.5 错误：用普通输入框方式覆盖 SQL

问题：

- SQL 页使用的是 Ace 编辑器
- 直接 `fill` 或键盘输入容易拼接旧内容

正确办法：

- 直接用 Ace API 写入：
  - `window.ace.edit('ta-sql-ace-editor').setValue(sql, -1)`

### 6.6 错误：假设所有项目都有 `bundle_id`

问题：

- `171` 已验证不是 `bundle_id` 型项目

正确办法：

- 每个项目先做字段探查
- 不跨项目硬套字段

### 6.7 错误：把现成业务指标当成裸 SQL 指标

问题：

- 例如“新增玩家”“次留”等报表值，可能使用系统口径、隐藏过滤或预计算

正确办法：

- 先用报表定义校准一次
- 口径对齐后再固化 SQL 模板

## 9. 性能与部署建议

### 9.1 当前实测时延

已实测接口层时延：

- `GET /v1/ta/space/getSpaceTree?projectId=170`
  - `time_total ≈ 0.207s`
- `POST /v1/ta/dashboard/search-dashboard?...`
  - `time_total ≈ 0.077s`
- `GET /v1/ta/entity/listEntities?projectId=170`
  - `time_total ≈ 0.105s`

结论：

- 后台配置接口本身很快。
- 真正慢的部分不在接口层，而在：
  - 浏览器切页
  - 前端渲染
  - Playwright 页面操作
  - SQL 执行等待
  - CSV 下载

### 9.2 当前链路中最慢的环节

从当前实测看，慢点主要有四类：

1. 浏览器导航与切项目
   - 常见在 `2s ~ 5s`
2. SQL 页前端操作
   - 写入 SQL、点击计算、等待按钮状态恢复
3. SQL 引擎执行
   - 轻量查询通常几秒
   - 重查询可能几十秒甚至失败
4. CSV 下载
   - 通常 `1s ~ 3s`

### 9.3 当前不适合部署的办法

以下办法不适合后续部署或定时化：

- 靠页面可见文本抄数
- 依赖频繁切项目、切页面、等页面加载
- 让浏览器承担结构探查和结果提取主任务

原因：

- 稳定性差
- 时延高
- 难并发
- 很难扩展到批量任务

### 9.4 更高效的固定办法

后续应优先采用三层结构：

1. 接口层
   - 直接请求：
     - 项目空间树
     - 看板定义
     - 报表定义
     - 项目元信息
   - 用于：
     - 找项目
     - 找看板
     - 找报表
     - 找口径

2. SQL 层
   - 直接执行轻量 `SPL/SQL`
   - 用于：
     - 项目探查
     - 产品识别
     - 正式数据查询

3. 浏览器层
   - 只保留：
     - 登录
     - 少量口径校准
     - 找不到接口时的备用路径

### 9.5 当前建议的加速方向

优先级从高到低：

1. 把“项目空间树、看板树、报表树”全部改成直连接口
2. 把“项目识别映射”固化成本地库，避免重复探查
3. 为每个项目建立固定 SQL 模板
4. 对 SQL 查询增加轻量优先策略
5. 给每一步打点计时，输出耗时日志

### 9.6 SQL 性能固定规则

- 优先单表查询，不先联表
- 优先查 `v_user_<projectId>` 做结构探查
- 查事件表默认带 `"$part_date"`
- 一次只验证一个字段或一个口径
- 重查询拆成多条轻查询

### 9.7 必须加入的时间测试点

后续正式脚本必须记录这些耗时：

- 登录态准备耗时
- 项目探查耗时
- 空间树接口耗时
- 看板定义接口耗时
- 报表定义接口耗时
- SQL 执行耗时
- CSV 下载耗时
- 本地解析耗时

### 9.8 部署版推荐目标

部署版目标应改成：

- 配置探查全部走接口
- 正式取数全部走 SQL
- 浏览器只做登录态初始化和少量人工校准

如果后面继续优化，优先继续找：

- SQL 执行对应的后台接口
- 能否直接跳过浏览器页内点击 `计算` 和 `全量下载`
- 能否直接通过接口获取 SQL 查询结果文件或结果集
