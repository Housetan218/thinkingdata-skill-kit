# ThinkingData 登录到分析架构流程

更新时间：2026-04-03

> 目标：固定“登录后台 -> 取得登录态 -> 确认项目 -> 进入 SQL 链路”的最短安全路径。

## 1. 固定入口

- 登录页：`https://taadmin.unbing.cn/oauth/#/login`
- 后台主站：`https://taadmin.unbing.cn`
- SQL 页：`https://taadmin.unbing.cn/#/tga/ide/-1?tab=result`

## 2. 标准流程

1. 在任一可复用的已登录浏览器会话中打开后台并确认已登录。
2. 确认后台首页可见，且左上角项目切换器可以打开。
3. 浏览器自动化默认优先使用 Playwright CLI 来复用当前已登录浏览器状态，或接管当前会话。
4. Browser Use CLI 只在 Playwright 不稳定或只需要轻量页面恢复时作为 fallback 使用。
5. 在后台页面读取 `localStorage`：
   - `ACCESS_TOKEN`
   - `REFRESH_TOKEN`
   - `TA_MASTER_PROPS`
   - `TA_AUTHS`
6. 后续接口统一使用：

```http
Authorization: Bearer <ACCESS_TOKEN>
```

7. 先确认项目，再决定是否需要探表。
8. 已建档项目直接走 SQL 模板；新项目先探结构再写 SQL。
9. SQL 优先走 WebSocket 直连；浏览器 SQL 页只做回退。

## 2.0 登录态判断补充规则

不要把“Playwright 默认空白自动化会话未登录”直接等同于“当前不存在可复用已登录会话”。

必须区分三件事：

1. 默认自动化会话是否已登录
2. 是否存在可复用的真实浏览器已登录会话
3. 当前自动化是否已经成功接管该已登录会话

只有在这三项都检查后，才能下“当前无法复用登录态”的结论。

## 2.0.1 会话分裂模型

要默认接受下面这个事实：

- Playwright 默认自动化会话
- 真实 Chrome profile 持久化资料
- 当前 live 已登录页面会话

这三者可能彼此分裂，不自动等价。

因此：

- 不能用默认自动化会话未登录，推导真实浏览器一定未登录
- 不能用真实浏览器残留 token，推导当前 live 会话一定有效
- 最终只信当前 live 已登录页面里重新读取到的鉴权状态

## 2.1 登录失效时的停止规则

如果当前 live 页面已经落到登录页，或当前 live 会话里的鉴权校验明确失败：

1. 停止正式查询路径
2. 立即在真实浏览器打开 `https://taadmin.unbing.cn/oauth/#/login`
3. 明确告诉用户“当前登录态失效，请在已弹出的登录页完成登录”
4. 登录恢复后再继续项目确认和 SQL

不要把历史 localStorage 残留里的多个 `ACCESS_TOKEN` 当成默认恢复手段去逐个探测。那只适合调试，不适合正式查询流程。

也不要把复制 Chrome profile、无头接管 profile、或 `--save-storage` 是否产出文件，当成正式恢复链路的成功判据。这些都只算辅助尝试，不算标准恢复路径。

## 2.1 执行闸门

只有以下信息明确后，才允许进入 SQL：

- 项目或产品范围
- 日期或日期范围
- 指标或分析问题
- 输出格式

如果上面有任何一项不明确，固定动作是：

1. 只做登录态检查
2. 只做项目确认
3. 说明缺失项
4. 停止继续执行，等待下一条明确指令

不要为了“继续推进”就先打开 SQL 页。

## 3. 读取登录态的固定方式

在后台 shell 页面执行：

```js
localStorage.getItem('ACCESS_TOKEN')
localStorage.getItem('LOGIN_NAME')
```

固定原则：

- token 只用于当前会话，不写入仓库
- token 失效就重新从已登录页面读取
- 不要把 token 抄进长期文档
- 工具输出如果带 `KEY=value` 包装、转义噪音、或 transport 包装，先清洗再用于鉴权

## 3.1 当前已验证的恢复方式

目前最稳的恢复方式只有这一条：

1. Playwright 打开真实登录页
2. 用户在真实页面完成登录
3. 从当前 live 会话重新读取 `ACCESS_TOKEN / REFRESH_TOKEN / TA_MASTER_PROPS / TA_AUTHS / LOGIN_NAME`
4. 用当前 live 鉴权状态继续项目确认与 SQL

目前还没有一条已验证的“只靠 `REFRESH_TOKEN` 自动续期”的标准恢复动作，因此不要把它写成默认流程。

## 4. 项目确认规则

- 先确认项目 ID，再跑 SQL
- 单产品项目可以直接跑基础模板
- 聚合项目必须先拆产品
- 特殊项目不要默认套 `bundle_id`

当前常用映射提示：

- `123` = LiCoWa iOS
- `125` = LiCoWa Android
- `160` / `170` = 聚合项目
- `171` = 特殊项目

这些只是当前已建档样例，不代表后续同事只会操作这些项目。
正式查询前，仍然必须先确认当次真实项目 ID。

项目列表确认优先顺序：

1. 鉴权后的项目列表接口
2. 其次才是页面项目切换器

原因：

- 顶部项目切换器 DOM 交互不稳定
- 已鉴权接口对“当前可见项目列表”更可靠

## 5. SQL 进入规则

优先顺序：

1. 已建档项目 + 已有模板：
   - 直接 WebSocket 直连
2. 新项目或异常项目：
   - 先探结构
   - 再写 SQL
3. 直连受阻：
   - 回退到浏览器 SQL 页

进入 SQL 前，至少要明确：

- `projectId`
- 日期或日期范围
- 目标指标
- 结果要按什么格式给出

如果用户只说“开始分析”或“帮我看看数据”，先停在项目确认层，不要直接跳到 SQL。

详见：

- [data-acquisition-methods.md](data-acquisition-methods.md)

## 6. 常见错误

- 登录后直接看仪表盘读数
- 未确认项目就开始跑 SQL
- token 没更新，导致后台接口误报未登录
- 聚合项目没拆产品就直接输出结论
- 没有明确任务就先打开 SQL 页
- 写 SQL、计算、下载 CSV

还不稳定的部分：

- 不同项目的“正试用户”口径字段不一致
- 留存和使用率指标还需要逐项目校准
- 数数查询引擎全局队列会显著影响耗时

## 7. 推荐团队使用方式

团队初版 skill 只负责：

1. 登录和鉴权检查
2. 项目结构探查
3. 项目识别字段确认
4. SQL 页标准操作
5. 常见错误拦截

先不要在初版 skill 里承诺：

- 所有产品可直接一键出完整日报
- 所有比率指标都天然口径一致
- 无需人工校准

## 8. 同事最短启动口令

推荐同事这样启动：

```text
请使用 thinkingdata-analysis-entry skill。先只检查登录态和项目列表，不要打开 SQL；等我给你项目、日期、指标后再执行查询。
```

如果已经知道目标，也可以直接说：

```text
请使用 thinkingdata-analysis-entry skill。项目 <projectId>，日期 2026-04-01，指标是新增和活跃，按日报格式输出；优先走 WebSocket SQL 直连。
```

## 9. 故障提示：打开 SQL 页后长时间没有结果

常见根因：

- 指令里没有写清项目、日期、指标
- skill 误把“开始分析”理解成“立刻执行 SQL”
- 进入了浏览器 SQL 页而不是 WebSocket 直连

处理顺序：

1. 退出当前发散流程
2. 回到项目确认
3. 补齐项目、日期、指标、输出格式
4. 优先改走 WebSocket 直连
