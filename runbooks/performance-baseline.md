# ThinkingData 性能基线

更新时间：2026-04-03

> 用途：让使用 skill 的同事知道“现在为什么快了、慢点还在哪里”。

## 1. 已确认的瓶颈变化

旧链路主要耗时：

1. 打开 SQL 页
2. 写 Ace 编辑器
3. 点计算
4. 轮询页面状态
5. 点下载 CSV
6. 再读结果

新链路主要耗时：

1. WebSocket 建连
2. 服务端排队
3. SQL 引擎执行

结论：

- 浏览器页面已经不是主瓶颈
- 当前最大波动来自服务端排队和执行

## 2. 已验证的改善幅度

一个 7 项指标样本：

- 旧浏览器链路总耗时：`106.135s`
- 新 WebSocket 直连总耗时：`19.573s`

改善结果：

- 减少 `86.562s`
- 下降约 `81.6%`
- 速度约 `5.4x`

## 3. 直连波动样本

同一类 SQL 直连也会波动：

- 低排队样本：
  - 总耗时 `4.914s`
  - 引擎耗时 `709ms`
- 高排队样本：
  - 总耗时 `39.760s`
  - 引擎耗时 `39499ms`

所以现在的真实判断应该是：

- 直连已经把浏览器冗余耗时打掉了
- 但服务端排队仍会让总耗时上下波动

## 4. 当前建议

如果要继续优化，优先顺序是：

1. 合并 SQL，减少查询次数
2. 把常用指标都迁移到直连脚本
3. 观察不同时间段的排队情况
4. 避免回退到浏览器读表模式
