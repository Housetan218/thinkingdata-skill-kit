# ThinkingData 项目初始化标准

更新时间：2026-04-03

> 目标：把“一个项目第一次接入分析流程时要做什么、要记录什么、以后怎么复用”固定下来，避免每次重新探路径、重新猜字段、重新解释产品。

## 1. 这份文档解决什么问题

项目初始化不是“先随便跑几条 SQL 看看”。

项目初始化的目标是一次性固化下面四类信息：

1. 登录和会话复用方式
2. 项目结构、表结构、字段结构
3. 产品映射、口径、输出模板
4. 项目特例、特殊字段、特殊计算方式

初始化完成后，日常查询默认不再重做全量探查，而是直接复用本地记录。

固定落地目录：

- 项目档案总入口：`<analysis-root>/projects/README.md`
- 项目模板目录：`<analysis-root>/projects/_template/`
- 项目正式档案目录：`<analysis-root>/projects/<projectId>/`

其中 `<analysis-root>` 表示你团队本地用于存放 ThinkingData 资料与项目档案的工作目录。

## 2. 什么时候需要初始化

以下场景必须做一次正式初始化：

- 新项目第一次接手
- 已知项目，但没有本地项目档案
- 聚合项目第一次拆产品
- 项目字段明显变化
- 历史模板已明显失真

以下场景不需要重做初始化：

- 已建档项目的日常日报、周报、临时查询
- 仅切换日期、国家、渠道等普通筛选
- 只是在既有模板上补一两个常规指标
- 只是因为登录态过期而重新登录

## 2.1 什么时候只需要局部探表

以下场景通常只需要局部探表，不需要重做全量初始化：

- 只缺某个指标的字段或事件定义
- 只缺子产品映射
- 只缺正式用户过滤口径
- 只缺 country / platform / app version / event time 这类单一核心字段确认
- 用户明确要求“先重探这一项”

原则：

- 能局部探表就不要全量重探
- 登录恢复后先读 dossier，再决定是否需要局部探表
- 用户可以主动要求局部或全量重探

## 3. 初始化完成后的固定产物

每个项目初始化完成后，本地至少要有下面四类产物。

### 3.1 项目档案

建议每个项目建立一份独立项目档案，内容至少包括：

- 项目 ID
- 项目名称
- 项目类型
  - 单产品项目
  - 聚合项目
  - 特殊项目
- 项目常用时区
- 是否已验证登录态复用
- 用户表 / 事件表
- 主识别字段
- 产品映射关系
- 常用过滤条件
- 已确认的异常点

推荐固定为四份文件：

- `overview.md`
- `fields.md`
- `metrics.md`
- `output-template.md`
- `query-templates.md`

### 3.2 字段字典

字段字典不是简单列字段名，而是要写清：

- 字段名
- 所在表
- 字段含义
- 典型值
- 是否主识别字段
- 是否业务强依赖字段
- 是否为项目特有字段
- 验证日期
- 验证 SQL 或验证来源

### 3.3 口径与模板库

至少要登记：

- 新增口径
- 活跃口径
- 留存口径
- 使用率口径
- 保存率口径
- 国家分布口径

每条口径都要写清：

- 指标名称
- SQL 模板位置
- 依赖字段
- 过滤条件
- 是否已和看板校准
- 最近校准日期

### 3.4 输出模板

每个项目要固定至少一种“标准输出格式”。

不能只输出一段纯文字。至少要有：

- 标题块
- 数据块
- 简析块
- 风险或备注块

## 4. 初始化时的标准步骤

### 4.1 登录与长期复用登录态

初始化时先确认：

- 后台是否能正常登录
- 当前浏览器是否能长期维持登录
- 自动化是否能复用当前登录态
- 是否已经能稳定读取 `ACCESS_TOKEN`

要记录的是“登录态获取办法”，不是记录真实 token。

初始化档案中要写清：

- 已验证浏览器会话类型
- 推荐复用方式
  - 已登录浏览器会话接管
  - cookie 导入
  - 已登录页面读取 `localStorage`
- 失效后的重取路径
- 最近验证日期

不要把下面内容写入文档：

- `ACCESS_TOKEN` 真值
- `REFRESH_TOKEN` 真值
- 任何账号密码

Browser Use CLI 在这一段只负责：

- 检查登录态是否还有效
- 打开或恢复后台页 / SQL 页
- 确认当前页面确实处在目标项目上下文

Browser Use CLI 不负责：

- 代替 SQL 主链路
- 代替口径确认
- 缩短服务端 SQL 耗时

### 4.2 结构探查

初始化探查只做一次标准化探查，目标不是“尽可能多点几下”，而是把后续复用所需信息探出来。

至少要确认：

- `v_user_<projectId>` 是否可用
- `v_event_<projectId>` 是否可用
- 主时间字段
- 可用于拆产品的字段
- 可用于识别测试 / 正试用户的字段
- 是否存在项目特有业务字段

聚合项目必须额外确认：

- 产品拆分字段
- 每个产品的主值
- 是否存在历史脏值、测试值、渠道值

特殊项目必须额外确认：

- 不能沿用的默认字段
- 需要改用的替代字段
- 该替代字段的稳定性

### 4.3 建立项目认知

项目初始化不只是结构探查，还要补产品认知。

每个项目档案要至少写清：

- 产品中文名 / 英文名
- iOS / Android / 聚合归属
- 主产品包名或产品识别值
- 产品主要功能
- 关键行为事件
- 重点国家或市场
- 当前常看的核心指标

这样后面跑出来的数据，才能附带最基础的解释，而不是只有数字。

### 4.4 固化模板

初始化完成后，要把下面内容落到本地模板里：

- 基础新增 SQL
- 基础活跃 SQL
- 基础次留 SQL
- 常用使用率 SQL
- 常用保存率 SQL
- 国家 TopN SQL
- 可合并的多指标汇总 SQL

原则：

- 一项目一组模板
- 聚合项目按产品拆模板
- 特殊项目单独写模板，不跨项目硬套
- 模板记录最近校准日期
- 模板记录是否已和可信输出核对

## 6. 特殊字段如何补充与调用

任何项目特有字段，一律不要只留在记忆里。

必须落入项目档案，且至少写清：

- 字段名
- 表名
- 字段解释
- 它和默认字段有什么不同
- 在什么场景必须用它
- 对应 SQL 片段
- 最近一次验证时间

推荐记录方式：

```md
### 项目特有字段：`account_platform`

- 所在表：`v_user_171`
- 用途：项目 171 中替代默认产品识别字段
- 不能使用：`bundle_id`
- 推荐用法：
  - 先按 `account_platform` 做分组探查
  - 再结合 `campaign_name` / `media_source` 二次确认
- 最近验证日期：2026-04-03
```

调用原则：

- 优先读项目档案里的特有字段登记
- 没登记的字段，不要直接进入正式口径
- 若临时发现新字段，先补档案，再给正式结果

## 7. 特殊计算方式如何记录

有些项目的特殊性不在字段，而在计算逻辑。

比如：

- 次留不是标准新增口径
- 保存率分母不是标准新用户
- 使用率需要排除某类测试用户
- 国家分布要额外剔除某些包

这类情况必须单独登记到项目档案的“口径与特殊计算”章节，至少写清：

- 指标名称
- 默认算法
- 项目特殊算法
- 特殊原因
- 对应 SQL 模板
- 是否已经与看板或业务方校准

如果没有登记，不允许口头默认“这个项目也差不多”。

## 8. 标准输出格式

以后查询结果，推荐至少按下面四段输出。

### 8.1 数据摘要块

```text
《<产品名>》 <timezone>
<日期>
新增 <值>，活跃 <值>
前一日次留 <值>%
<指标 A> <值>%
<指标 B> <值>%
<指标 C> <值>%
新增前五分布：<国家1 值1> <国家2 值2> <国家3 值3> <国家4 值4> <国家5 值5>
```

### 8.2 初步分析块

至少补 2 到 4 条简析，例如：

- 新增主要集中在东南亚市场，印度尼西亚仍是第一来源。
- 活跃相对新增倍数稳定，说明短期活跃盘没有异常塌缩。
- 拍照屋使用率明显高于照片条保存率，说明使用到保存之间仍有转化损失。

### 8.3 项目认知块

用于补充项目背景：

- 该产品属于什么业务类型
- 当前主包或主识别值是什么
- 当前最关注的核心行为是什么

### 8.4 风险块

如果本次结果带有不稳定因素，必须明确写出：

- 是否使用了临时字段
- 是否使用了未校准指标
- 是否存在项目队列耗时过长
- 是否仍需和看板再核一次

## 9. 初始化之后，什么时候才允许重探

只有下面情况才允许回到重新探查：

- 表结构发生变化
- 主识别字段失效
- 产品主值变化
- SQL 和历史结果偏差异常
- 新增业务字段首次进入正式分析

其他日常查询默认直接复用：

- 项目档案
- 字段字典
- SQL 模板
- 输出模板

## 11. 固定结论

- 初始化的核心不是“跑通一次”，而是“以后不用重跑探查”
- 登录态要记录获取办法，不记录敏感值
- 项目字段要形成档案，不靠记忆
- 特殊计算必须落文档，不允许口头继承
- 输出结果必须包含数据、简析、产品认知、风险说明
